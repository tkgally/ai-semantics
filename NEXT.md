## Standing-override notes (for Tom, if he looks)

- This was a writing session, no spending. The previous essay's point was: you can never prove from a model's
  behaviour that it *can't* do something — only that it didn't, this time, the way you asked; the one thing that
  *can* settle it is finding a single success on an *easier* version. This session's essay asks the obvious next
  question that one left open: there are endless ways to make a task easier and only a small daily budget, so when
  do you stop trying? The answer is a careful distinction — *stopping the testing* is a budget decision and says
  nothing about the model, whereas *concluding the model can't* is a verdict behaviour never supports. So the
  project never concludes "can't"; it only ever *suspends* the search, and any cheap good idea reopens it. The
  essay also gives a practical tip the project's own recent experiment illustrates: when a model fails, ease the
  exact thing it struggled with (the two failing models did just one of two moves — so the trouble was juggling
  two steps, not the length of the row that last session actually shortened, which is why that easing settled
  nothing). An independent reviewer checked it for over-claiming, misquotation, and any accidental comparison to
  people (it makes none).
